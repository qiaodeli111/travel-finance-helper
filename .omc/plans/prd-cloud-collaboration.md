# PRD: 多人协作与云同步功能

## 文档信息
- **创建日期**: 2026-03-01
- **项目名称**: 旅行分账助手 (Travel Finance Helper)
- **功能模块**: 多人协作/云同步

---

## 1. 问题陈述

### 1.1 当前痛点
旅行分账助手目前是一个纯本地应用，存在以下问题：

1. **数据孤岛**: 数据仅存储在单一设备本地存储中，无法跨设备访问
2. **协作困难**: 多家庭出行时，只能由一人负责记录，其他人无法实时查看或参与记账
3. **数据丢失风险**: 依赖本地存储，设备丢失或清除浏览器数据会导致账本永久丢失
4. **分享不便**: 结算方案只能通过导出文件分享，无法实时同步最新状态

### 1.2 目标用户
- 多家庭结伴出行的用户群体
- 需要跨设备访问账本的用户
- 希望多人共同参与记账的团队

---

## 2. 目标与非目标

### 2.1 目标 (Goals)
1. **实时协作**: 多用户可同时编辑同一账本，实时看到彼此的更新
2. **云端存储**: 数据安全存储在云端，支持跨设备访问
3. **用户系统**: 提供完整的账户注册、登录、管理功能
4. **便捷分享**: 通过链接或二维码快速邀请他人加入账本

### 2.2 非目标 (Non-Goals)
- 不构建复杂的社交功能（好友系统、动态等）
- 不支持多语言协作（当前已有多语言支持，保持现状）
- 不提供支付或转账功能
- 不做商业化运营（会员、付费功能等）

## 3. 用户故事与验收标准

### 3.1 用户账户系统

#### US-1: 用户注册
**作为** 新用户，**我希望** 能够通过邮箱密码或手机验证码注册账户，**以便** 在云端保存我的数据。

**验收标准:**
- [ ] 支持邮箱+密码注册，密码需满足安全要求（至少8位，包含数字和字母）
- [ ] 支持手机号+验证码注册（需集成短信服务）
- [ ] 注册成功后自动创建默认账本
- [ ] 注册时显示服务条款和隐私政策

#### US-2: 用户登录
**作为** 已注册用户，**我希望** 能够通过多种方式登录，**以便** 访问我的云端数据。

**验收标准:**
- [ ] 支持邮箱+密码登录
- [ ] 支持手机号+验证码登录
- [ ] 支持匿名/游客模式（本地数据，不与云端同步）
- [ ] 登录失败时显示清晰的错误提示
- [ ] 支持"记住我"功能（保持登录状态30天）
- [ ] 支持退出登录功能

#### US-3: 账户管理
**作为** 用户，**我希望** 能够管理我的账户信息，**以便** 保持账户安全和信息更新。

**验收标准:**
- [ ] 用户可查看和修改昵称、头像
- [ ] 用户可修改密码（需验证旧密码）
- [ ] 用户可绑定/解绑手机号和邮箱
- [ ] 用户可删除账户（需二次确认，删除后数据不可恢复）

### 3.2 云端数据存储

#### US-4: 数据同步
**作为** 登录用户，**我希望** 我的账本数据自动同步到云端，**以便** 在其他设备上访问。

**验收标准:**
- [ ] 登录后，本地数据自动上传到云端
- [ ] 打开应用时，自动拉取云端最新数据
- [ ] 数据变更后，自动同步到云端（增量同步）
- [ ] 离线时可继续使用，联网后自动同步
- [ ] 同步冲突时，显示冲突解决选项

#### US-5: 数据迁移
**作为** 从本地模式迁移的用户，**我希望** 能够将本地数据导入云端，**以便** 不丢失历史记录。

**验收标准:**
- [ ] 新登录用户可选择将本地数据迁移到云端
- [ ] 迁移过程显示进度
- [ ] 迁移完成后，本地数据保留为备份

### 3.3 多人实时协作

#### US-6: 账本成员管理
**作为** 账本创建者，**我希望** 能够邀请他人加入我的账本，**以便** 共同记账。

**验收标准:**
- [ ] 账本有明确的"创建者"和"成员"角色
- [ ] 创建者可生成邀请链接/二维码
- [ ] 创建者可移除成员
- [ ] 成员可主动退出账本
- [ ] 成员列表显示所有参与者及其角色

#### US-7: 邀请加入
**作为** 被邀请者，**我希望** 能够通过链接或二维码快速加入账本，**以便** 参与协作。

**验收标准:**
- [ ] 点击邀请链接可直接加入账本（已登录用户）
- [ ] 未登录用户点击链接后，引导登录/注册后自动加入
- [ ] 邀请链接可设置有效期（24小时/7天/永久）
- [ ] 已加入账本的用户点击链接显示"已加入"提示

#### US-8: 实时协作编辑
**作为** 账本成员，**我希望** 能实时看到其他人的编辑，**以便** 了解最新账务状态。

**验收标准:**
- [ ] 账单添加后，所有成员在3秒内看到更新
- [ ] 账单删除后，所有成员在3秒内看到更新
- [ ] 设置变更后，所有成员在3秒内看到更新
- [ ] 显示"正在编辑"的实时状态指示器
- [ ] 使用WebSocket或类似技术实现实时通信

### 3.4 权限控制

#### US-9: 角色权限
**作为** 账本创建者，**我希望** 能够控制成员的权限，**以便** 管理账本安全。

**验收标准:**
- [ ] **创建者(Owner)**: 完全控制权限，包括删除账本、管理成员、所有编辑权限
- [ ] **管理员(Admin)**: 管理成员、编辑账单、修改设置
- [ ] **成员(Member)**: 添加/编辑/删除自己创建的账单
- [ ] **访客(Viewer)**: 只读权限，可导出结算方案

## 4. 技术约束与方案

### 4.1 技术栈要求
- **前端**: React 19 + TypeScript + Vite（保持现有技术栈）
- **后端**: Serverless 架构
- **推荐方案**: Firebase 或 Supabase

### 4.2 架构约束
1. 保持对现有 Netlify 部署的支持
2. 使用 BaaS (Backend as a Service) 减少后端开发复杂度
3. 保持离线优先的设计原则，联网后同步
4. 数据库设计需支持实时订阅

### 4.3 推荐技术选型

#### Firebase 方案
```
- Authentication: 邮箱密码、手机验证码、匿名登录
- Firestore: 实时数据库，支持离线持久化
- Cloud Functions: 后端逻辑（可选）
- Hosting: 可继续使用 Netlify 或迁移到 Firebase Hosting
```

#### Supabase 方案（备选）
```
- Auth: 多种登录方式
- PostgreSQL: 关系型数据库 + Realtime
- Storage: 文件存储
- Edge Functions: 后端逻辑
```

## 7. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Firebase/Supabase 服务在国内访问不稳定 | 高 | 提供离线模式，考虑国内云服务备选方案 |
| 短信验证码服务成本 | 中 | 优先实现邮箱登录，手机登录作为增强功能 |
| 实时同步冲突处理复杂 | 中 | 采用乐观锁 + 最后写入胜出策略 |
| 用户隐私数据安全 | 高 | 数据加密存储，遵守相关法规 |
